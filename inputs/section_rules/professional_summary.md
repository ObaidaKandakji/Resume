# Professional Summary Rules

## Purpose

Write a short, role-specific summary that helps a recruiter understand fit in a few seconds.
If the summary cannot add clarity without sounding generic, keep it minimal and do not pad it with filler.

## Rewrite Policy

- The summary may be rewritten more freely than bullets when needed to remove generic language or align to the role.
- Even when rewritten, every claim must stay truthful and be supported by the resume, project specs, or user-provided background.

## Hard Constraints

- 2-3 sentences
- Roughly 35-60 words total
- No first person pronouns:
  - `I`
  - `me`
  - `my`
  - `we`
  - `our`
- No cliches or empty traits:
  - `hard-working`
  - `passionate`
  - `team player`
  - `fast learner`
  - `results-driven`
  - `seeking an opportunity`
  - `eager to contribute`
  - `leverage my skills`
- Include the target role title from the job description when practical.
- If the exact title is awkward or inflated, use the closest truthful normalized title.
- Include 2-3 exact job-description keywords only if they are supported elsewhere in the resume or project specs.
- Include 1 concrete proof point:
  - metric if available
  - otherwise concrete scope such as number of services, systems, workflows, or team size
- Mention only tools, domains, or claims that appear elsewhere in the resume or project specs.
- Do not list more than 3 technical strengths in the summary.
- Do not turn the summary into a tools dump.
- Do not repeat a generic objective statement.

## Preferred Structure

- Sentence 1:
  - identity
  - target role
  - strongest technical fit
- Sentence 2:
  - strongest project, internship, or experience proof point
  - measurable result or concrete scope
- Sentence 3 if needed:
  - teamwork, problem-solving, or delivery style
  - clear value to the team

If using only 2 sentences, combine the work-style/value signal into sentence 2 without losing specificity.

## Quality Bar

- Target a clear lane when possible:
  - backend
  - cloud
  - full-stack
  - data
  - embedded
  - platform
- Use job-description language naturally, not as keyword stuffing.
- Prioritize what a recruiter cannot infer instantly from the skills list alone:
  - role fit
  - strongest relevant proof
  - working style or delivery value
- Prefer concise nouns and verbs over polished but empty phrasing.
- Avoid long comma-chains of technologies.
- Avoid broad motivational language.
- Avoid repeating obvious facts already visible in Education unless they sharpen fit for the role.

## Final Self-Check

Before accepting a rewritten summary, confirm:

- It sounds specific rather than polished-and-empty.
- Each tool or domain named can be pointed to elsewhere in the resume or project specs.
- It contains a real proof point, not only traits.
- It can be scanned quickly in one pass.
